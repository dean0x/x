# LinkedIn Thought Piece Template

## Structure
A genuine opinion or lesson, backed by your real experience. Not a motivational poster.

## Format
```
[Contrarian or surprising opening line]

[Your story / evidence — what happened to make you think this]

[The actual insight — specific, not generic]

[Optional: question for discussion — but make it specific, not "What do you think?"]
```

## Rules
- Have an actual opinion. Lukewarm takes don't spread.
- Back it up with something you actually did or saw. Not theory.
- Don't write advice you haven't followed yourself.
- If you can replace your name with anyone else's and the post still works, it's too generic.
- Keep it under 1300 characters for optimal reach (LinkedIn truncates at ~1300).

## Good Example
```
I stopped doing code reviews line-by-line. Here's what I do instead.

For 3 years I'd go through every PR file by file, leaving comments on style, naming, edge cases. It took hours and most of my comments were nitpicks.

Now I read the PR description, check out the branch, and actually run it. I spend 10 minutes using the feature. Then I write one comment: either "this works, ship it" or "here's what broke when I tried X."

My review time dropped from 45 minutes to 15. And the reviews are more useful — I'm catching real bugs instead of arguing about variable names.

The linter handles style. Tests handle edge cases. My job is to answer one question: does this actually work?

What's your least popular code review opinion?
```

## Bad Example
```
5 Lessons I Learned About Code Reviews:

1. Be respectful
2. Focus on the big picture
3. Don't nitpick
4. Be timely
5. Learn from others

What would you add to this list? [GENERIC, NO STORY, LISTICLE]
```
